AGREE defensive heuristic rules.

1. Declare internal AGREE variables explicitly with `eq` and a type before use.
2. Do not redeclare AADL feature names as AGREE `eq` variables.
3. Define primitive aliases such as `int`, `real`, and `bool` at the package level when they are used.
4. Add a matching `with` clause for every package-qualified reference such as `Base_Types::Float`.
5. Put `eq`, `const`, `assume`, and `guarantee` clauses in component type annexes when they describe interface behavior.
6. Put implementation-oriented `assign` clauses in component implementation annexes.
7. The left-hand side of `assign` must already exist as an output feature or declared auxiliary variable.
8. Do not define the same variable with both an initialized `eq` expression and an implementation-level `assign`.
9. Use AGREE logical operators: `and`, `or`, and `not (...)`.
10. Do not use C/Java-style logical operators such as `&&`, `||`, or `!`.
11. Use `=` rather than `:=`.
12. Comparison expressions return `bool`; do not assign them to numeric variables.
13. Declare fixed thresholds with `const`.
14. Define `pre()` recurrences with the `->` operator in a type-level `eq`.
15. Use `pre(x)` with parentheses.
16. Use unique quoted labels for labeled `assume`, `guarantee`, `assert`, and `assign` clauses.
17. Avoid non-ASCII mathematical or logical symbols in generated AADL+AGREE artifacts.
18. Do not call unsupported math-library functions unless they are explicitly provided by a referenced node or package.
19. Preserve the full requirement chain: trigger, constraint, and required response.
20. Use one-way implication `=>` unless the requirement explicitly states equivalence.
21. Avoid unsupported temporal liveness operators in this validator setup.
22. Ground every AGREE variable in the current AADL model or an explicit declaration.
23. Do not reference implementation-only subcomponent paths directly from an incompatible component type scope.
24. Treat each `annex agree` block as scoped to its owning component declaration.
25. Use implementation-level assignments to bind subcomponent outputs when needed.
26. Prefer nested `if ... then ... else` expressions for branching logic.
27. Do not use unsupported `elsif` syntax.
